# 與使用者溝通原則
* 你生成的 Implementation Plan、TODO List 等，需要我 review 的，請一律用中文撰寫再給我確認
* 當你向我詢問，或者回應我的訊息時，始終用中文和我對談 (你自己的思考過程不在此限)


# 規劃原則 (Planning)
* 除小問題、小調整或我明確說不用 plan 之外，任何工作都要先規劃並與我對齊
* 你必須等我 review 完整個 plan，並明確的說 "開始" 才能開始實作，否則視為我不同意。唯一例外是我明確告訴你 "plan 完就直接做"
* plan 的細節要充足，你要預期這份 plan 可以交給其他人，他們只需要依循著文件去做
* 規劃過程，有任何你不確定需要我裁決的，你先選擇一個你推薦的方案，並且在 plan 的最後一個 section 標記這些問題，以及你推薦的答案，並在規劃完畢後明確向我提及有哪些問題

## 流程
1. 載入 `agent-skills/planning-and-task-breakdown`，進入唯讀模式，不寫任何程式
2. 將 plan 與 task 清單持久化成檔案，放在 `.planning/YYYY-MM-DD-<slug>/` 底下(你自己建立目錄)
   (YYYY-MM-DD 為當日日期，slug 為此次工作的簡短英文代稱):
   - 實作計畫: `.planning/YYYY-MM-DD-<slug>/plan.md`
   - 任務清單: `.planning/YYYY-MM-DD-<slug>/todo.md`
   - 規格(如有): `.planning/YYYY-MM-DD-<slug>/spec.md`
3. 每個 task 需含:描述、驗收條件、驗證方式、相依、預估規模
4. plan 與 todo 寫好後，用中文向我摘要，若有需要我決定的問題，你也順便在這提一下


# 開發原則 (Development)
* 你的 code 始終保持可讀性
* 不要寫一大堆註解，註解只有在解釋複雜邏輯時才會需要，但現有系統有註解，如果不是移除程式碼，請別動他，前人留在那的註解有其意義
* 不要塞一堆註解說明在 function 開頭，function 名稱應該要 self-explanatory，除非真的很複雜的邏輯或參數量非常多才需要註解
* 遵守軟體開發最佳原則，良好的模組化，並保持擴充性，但不要過度設計
* 確保每個 function、class 都有明確的責任，並且遵守單一職責原則
* 不要重造輪子，開發前務必確認現有的模組、utils、套件等，能夠 reuse 請務必要 reuse
* 你如果看到 editor 上的 lint 的提示或警告，請直接忽略，因為可以一鍵修正，只有在跑過腳本依然無法修正才手動修改
* 最小化修改原則，只修改當下需求所必要的程式碼，不要帶著「順便修改」的心態去改其他程式碼，除非是重構或修正 bug


# Commit 原則
* 每次有些進展就記得要 commit
* 一次只做一件事情，一個進度做完就先 commit，為了容易追蹤，所以我們定位一個 commit 對上一個功能 or 修正 or refactor 等
* commit 記得進行驗證，包含但不限於: lint、build、功能測試等，確保 commit 的程式碼是可運行的
* Commit header 盡量簡短，而且要有 type 開頭，範例 `feat: 新增 XXX API`，具體 commit type 如下:
  * feat: 新增功能、功能變更、UI 調整
  * fix: 修正 bug
  * refactor: 重構程式碼、改善程式碼品質
  * chore: 部署、CI/CD、專案設定等
  * docs: 文件、註解、README 等(修正 typo、敘述等也是這項)
* 在寫 fix 的 commit 時，
  * header: 以敘述問題的角度撰寫，例如: `fix: 無法開啟 XX 畫面`
  * message: 必須要寫 "原因" 和 "解法"，例如:
    ```
    問題: 因為 xxx 原因，所以狀態管理器沒有被初始化，導致無法開啟 XX 畫面
    解法: 在初始化時加入狀態管理器的初始化，確保能 ooo
    ```


# 安全性準則
以下「取得明確同意」的意思是「先告知我你即將做甚麼變更，然後要我明確同意說，或者我預先授權你要去做，你才能開始做」。

* 禁止使用 `git push`、`git merge`、`git rebase` 等任何可能會改變分支、歷史、遠端或覆蓋工作區的 git 指令，除非我明確要求你做。
* 禁止讀取、輸出、提交 `.env`、key、token、credential 等敏感資訊，但你可以寫腳本操作(例如從 .env 讀取後設置到環境變數等)，你寫的腳本確保你自己不會看到明碼即可
* 在修任何敏感資訊前，先取得明確同意，正式操作前要先備份，備份格式請用 `<file>.<ext>.bak-<number>`，例如 `.env.bak-1`
* 禁止在當前專案根目錄外進行破壞性操作，包含但不限於:刪除、移動、修改檔案或資料夾等，你必須取得明確同意


# 注意
* 不需要手動修復 coding style，專案通常具備一鍵修正的指令，只有在跑過腳本依然無法修正才手動修改
* 已經被 ignore 的文件，不要強制添加到 git
* 這個專案不需要寫測試(unit test、 integration test、e2e test 等)
* 撰寫任何 "會進 git 版控的文件"，都不要去引用 "不會進版控的文件"，例如: `README.md`(會進版控) 不要引用 `tmp/xxx.md`(沒進版控) 的文件
